# Исследование вариантов аутентификации для расширения Chrome (без бэкенда)

## Краткое резюме

Данный документ содержит полное исследование вариантов аутентификации для расширения Digital Zen Chrome с основным ограничением: **нельзя развернуть собственный бэкенд-сервер**.

> **📝 Примечание:** Полная версия исследования на английском языке доступна в файле `authentication-options.md`

---

## Текущая реализация

### Google OAuth ✅ (Реализовано)

- **Бэкенд:** ❌ Не требуется
- **Как работает:** Chrome Identity API (`chrome.identity.getAuthToken`)
- **Scopes:** `userinfo.email`, `userinfo.profile`
- **Статус:** Работает отлично, рекомендуется оставить

## Результаты исследования

### 1. GitHub Device Flow ⭐⭐⭐⭐⭐

**Нужен ли бэкенд?** ❌ Нет

**Поддержка PKCE/Device Flow?** ✅ Да (Device Flow)

**OAuth Scopes:**

- `user:email` - доступ к email
- `read:user` - чтение профиля пользователя
- `repo` (опционально) - для работы с репозиториями

**Влияние на UX:**

- Пользователь должен посетить GitHub и ввести код устройства
- Немного больше трения, чем обычный OAuth
- Лучшая безопасность для расширений

**Изменения в manifest.json:**

```json
{
  "permissions": ["identity"],
  "host_permissions": ["https://github.com/*", "https://api.github.com/*"]
}
```

**Redirect URI:**

- Не требуется (Device Flow не использует редиректы)

**Рекомендация:** ⭐⭐⭐⭐⭐ **Настоятельно рекомендуется**

- Идеально подходит для расширений Chrome
- Не требует бэкенда
- Хорошая модель безопасности
- Официальная поддержка GitHub

### 2. OAuth2 PKCE для различных провайдеров

#### 2.1 GitHub с PKCE

**Нужен ли бэкенд?** ❌ Нет

**Поддержка PKCE?** ⚠️ Ограниченная

**Рекомендация:** ❌ Используйте Device Flow вместо PKCE

- GitHub не полностью поддерживает PKCE для OAuth Apps
- Device Flow - лучший выбор для GitHub

#### 2.2 Microsoft (Azure AD / Microsoft Account)

**Нужен ли бэкенд?** ❌ Нет

**Поддержка PKCE?** ✅ Да (полная поддержка)

**OAuth Scopes:**

- `openid` - аутентификация OpenID Connect
- `profile` - базовая информация профиля
- `email` - адрес электронной почты
- `User.Read` - чтение профиля через Microsoft Graph

**Изменения в manifest.json:**

```json
{
  "permissions": ["identity"],
  "oauth2": {
    "client_id": "YOUR_MICROSOFT_CLIENT_ID",
    "scopes": ["openid", "profile", "email", "User.Read"]
  },
  "host_permissions": ["https://login.microsoftonline.com/*", "https://graph.microsoft.com/*"]
}
```

**Redirect URI:**

- Chrome extension: `https://<extension-id>.chromiumapp.org/`
- Должен быть зарегистрирован в Azure AD

**Влияние на UX:**

- Похоже на Google OAuth
- Пользователи авторизуются с учетной записью Microsoft
- Бесшовный опыт при использовании `chrome.identity.launchWebAuthFlow`

**Рекомендация:** ⭐⭐⭐⭐ **Рекомендуется**

- Хорошо для организаций, использующих Microsoft 365
- Сильная поддержка PKCE
- Широкая пользовательская база
- Требует более сложной реализации, чем Google

#### 2.3 GitLab с PKCE

**Нужен ли бэкенд?** ❌ Нет

**Поддержка PKCE?** ✅ Да (с GitLab 13.0)

**OAuth Scopes:**

- `read_user` - чтение информации пользователя
- `email` - доступ к email
- `openid` - OpenID Connect
- `profile` - профиль пользователя

**Изменения в manifest.json:**

```json
{
  "permissions": ["identity"],
  "host_permissions": ["https://gitlab.com/*"]
}
```

**Redirect URI:**

- Chrome extension: `https://<extension-id>.chromiumapp.org/`
- Должен быть зарегистрирован в настройках приложения GitLab

**Влияние на UX:**

- Чистый OAuth flow
- Пользователи авторизуются с учетной записью GitLab
- Хорошо для расширений, ориентированных на разработчиков

**Рекомендация:** ⭐⭐⭐⭐ **Рекомендуется**

- Отлично для инструментов разработчика
- Полная поддержка PKCE
- Возможность self-hosted
- Меньшая пользовательская база, чем GitHub/Microsoft

#### 2.4 Bitbucket с PKCE

**Нужен ли бэкенд?** ❌ Нет

**Поддержка PKCE?** ✅ Да

- Хорошо для пользователей экосистемы Atlassian
- Меньшая пользовательская база
- Полная поддержка PKCE

### 3. OAuth Authorization Code (традиционный)

**Нужен ли бэкенд?** ✅ ДА - **НЕ ПОДХОДИТ**

**Поддержка PKCE?** N/A

**Описание:**
Традиционный OAuth2 Authorization Code flow требует client secret, который нельзя безопасно хранить в расширении Chrome (код на стороне клиента можно инспектировать).

**Рекомендация:** ❌ **Не рекомендуется**

- Нарушает ограничение "без бэкенда"
- Риск безопасности при попытке реализации без бэкенда

### 4. OIDC провайдеры с PKCE

#### 4.1 Auth0

**Нужен ли бэкенд?** ❌ Нет (с PKCE)

**Поддержка PKCE?** ✅ Да (полная поддержка)

**OAuth Scopes:**

- `openid` - OpenID Connect
- `profile` - профиль пользователя
- `email` - адрес электронной почты

**Изменения в manifest.json:**

```json
{
  "permissions": ["identity"],
  "host_permissions": ["https://*.auth0.com/*"]
}
```

**Redirect URI:**

- Chrome extension: `https://<extension-id>.chromiumapp.org/`
- Должен быть настроен в панели Auth0

**Дополнительные возможности:**

- Социальные подключения (Google, GitHub, Facebook и т.д.)
- Корпоративные подключения (SAML, Azure AD)
- Многофакторная аутентификация
- Passwordless аутентификация
- Управление пользователями

**Влияние на UX:**

- Высоко настраиваемый интерфейс входа
- Может агрегировать несколько провайдеров идентификации
- Пользователи могут использовать социальный вход

**Рекомендация:** ⭐⭐⭐⭐⭐ **Настоятельно рекомендуется**

- Отлично для поддержки нескольких провайдеров
- Профессиональное управление идентификацией
- Доступен бесплатный тариф
- Отличная документация и SDK

### 5. Firebase Authentication

**Нужен ли бэкенд?** ❌ Нет (Firebase предоставляет бэкенд)

**Поддержка PKCE?** N/A (использует другой flow)

**Совместимость с расширениями Chrome:** ⚠️ **Ограниченная**

**Проблемы:**

1. Firebase Auth SDK ожидает веб-окружение с DOM
2. Popup аутентификация плохо работает в расширениях
3. Redirect-based auth конфликтует с архитектурой расширения
4. Chrome Identity API нельзя использовать с Firebase напрямую

**Обходные пути:**

- Требуют Cloud Functions (технически это бэкенд, хотя и serverless)
- Сложная настройка

**Изменения в manifest.json:**

```json
{
  "permissions": ["identity", "storage"],
  "host_permissions": ["https://*.firebaseapp.com/*", "https://*.googleapis.com/*"]
}
```

**Влияние на UX:**

- Может поддерживать несколько провайдеров через Firebase
- Сложная настройка для окружения расширения

**Рекомендация:** ⭐⭐ **Не рекомендуется для расширений**

- Разработан для веб-приложений, а не расширений
- Требует обходных путей и возможно Cloud Functions
- Есть лучшие альтернативы (Auth0, прямой OAuth)

### 6. WebAuthn / Passwordless

**Нужен ли бэкенд?** ⚠️ Частично (для хранения и проверки учетных данных)

**Поддержка PKCE?** N/A (другой протокол)

**Поддержка расширений Chrome:** ✅ Да (с ограничениями)

**Описание:**
WebAuthn - это стандарт W3C для passwordless аутентификации с использованием аппаратных токенов, биометрии или платформенных аутентификаторов.

**Требования к бэкенду:**

- Сервер должен генерировать challenge и проверять attestation/assertion
- Сервер должен хранить публичные ключи учетных данных

**Обходной путь для "без собственного бэкенда":**

- Используйте сервис вроде Auth0, Okta или Azure AD с поддержкой WebAuthn
- Они обрабатывают требования к бэкенду

**Изменения в manifest.json:**

```json
{
  "permissions": ["identity", "storage"]
}
```

**Влияние на UX:**

- Современный, безопасный пользовательский опыт
- Требует совместимое оборудование/платформу
- Не у всех пользователей есть WebAuthn-совместимые устройства

**Рекомендация:** ⭐⭐⭐ **Рекомендуется как дополнительный фактор**

- Использовать с Auth0/Okta/Azure AD
- Отлично для пользователей, заботящихся о безопасности
- Не подходит как основной/единственный метод аутентификации

## Архитектурное решение

### Вариант A: Прямые интеграции OAuth

- Реализовать Google (готово) + GitHub Device Flow + Microsoft PKCE
- Больше контроля, меньше зависимостей
- Больше кода для поддержки

### Вариант B: Использовать Auth0 как агрегатор

- Единая точка интеграции
- Поддержка Google, GitHub, Microsoft, GitLab и т.д.
- Меньше кода, больше функций
- Небольшая стоимость после бесплатного уровня

**Рекомендация:** Начать с **Варианта A** (прямые интеграции) для GitHub, затем оценить **Вариант B** (Auth0), если нужно больше провайдеров или расширенные функции.
